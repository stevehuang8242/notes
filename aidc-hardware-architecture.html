00 架構總覽 · Overview

一台被切成很多塊的，巨型電腦

傳統資料中心是「許多獨立伺服器的集合」；AI 資料中心則被設計成「一台被切成很多塊的巨型電腦」。理解這個差異，是理解所有下層硬體選擇的前提。

▸兩個維度：Scale-Up 與 Scale-Out

AI 訓練與推論的瓶頸幾乎都在資料移動，而非單純的算力。因此整套硬體圍繞兩種互相垂直的擴展方式設計：

Scale-Up · 縱向擴展

在單一機架內把多顆 GPU 透過極高頻寬、低延遲、且記憶體一致（coherent）的 fabric 連成「一顆邏輯上的大 GPU」。媒介是銅線（NVLink/NVSwitch），距離短、頻寬以 TB/s 計。world size 從 8 → 72 → 576 顆。

Scale-Out · 橫向擴展

把成千上萬個機架/節點連成叢集。媒介是光纖（InfiniBand / Ethernet），以 RDMA 搬移資料，頻寬以 Gb/s ~ Tb/s 計、延遲較高。負責跨節點的梯度同步與 checkpoint。

兩種互相垂直的擴展。左：scale-up 在一個機架內把 GPU 以 NVLink 全互連（all-to-all），銅纜、TB/s、記憶體一致——整櫃像一顆大 GPU。右：scale-out 把成千上萬個機架掛上 spine／leaf 交換層，光纖、RDMA，負責跨節點同步。GB200 NVL72 的突破，就是把左邊那個框從 8 顆一舉撐到 72 顆。

NVIDIA 的 GB200 NVL72 之所以是分水嶺，正是因為它把 scale-up domain 從一個機板的 8 顆 GPU，一舉拉到整個機架的 72 顆，讓 trillion-parameter 模型能「住在同一個 NVLink 空間裡」，把過去得靠 scale-out 網路承擔的同步流量，留在機架內的銅纜上解決。

▸由下而上的九個層次

本手冊的章節對應上方堆疊圖。從矽晶片往上，每一層都在解一個物理問題，而上層的能力受限於下層：

  • 矽 / 運算（§01）製程逼近光罩極限，被迫走向 chiplet 多晶片封裝；精度則一路往 FP8 / FP4 下探以換取吞吐。
  • 記憶體（§02）HBM 的容量與頻寬是大模型的硬牆，3D 堆疊與 CoWoS 封裝是當前主軸。
  • Scale-Up / Scale-Out 互連（§03–04）把運算單元連起來的兩張網，決定了叢集能多大、效率多高。
  • 機架系統（§05）把上述全部塞進一個機櫃的工程，密度爆增帶來重量、供電與散熱的連鎖限制。
  • 網路拓撲（§06）與儲存（§07）叢集級的連線方式與餵資料的管線。
  • 電力與散熱（§08–09）最終的物理天花板——一個機架現在動輒 100 kW 以上，氣冷已不可行。

「AI Factory」是 NVIDIA 用來描述這種設施的詞：輸入電力、矽與資料，持續輸出「token」。它把資料中心當成一條生產線來設計，而不是一間機房——這也是為什麼電力與散熱被當成第一級的設計參數，而非事後工程。

01 運算層 · Compute

做矩陣乘法的矽，一年一代往前推

GPU 是 AI 資料中心的引擎。要看懂規格表，得先掌握四個結構概念，以及一條貫穿三個世代的演進線。

▸架構基礎：四個關鍵概念

SM / Tensor Core

GPU 由數百個 Streaming Multiprocessor (SM) 組成，每個 SM 內含 CUDA Core（通用浮點）與 Tensor Core（矩陣乘加專用單元）。AI 算力幾乎全來自 Tensor Core，每一代都換新（Hopper 第 4 代 → Blackwell 第 5 代 → Rubin 第 6 代）。

Reticle Limit / Chiplet

單一晶粒的面積有光刻機的光罩極限。H100 已逼近此牆，因此 Blackwell 起改用雙晶粒（dual-die），以 NV-HBI 10 TB/s 連接成單一邏輯 GPU；Rubin 兩顆、Rubin Ultra 四顆 compute die。

Precision / 精度格式

從 FP64 一路到 FP8 / FP6 / FP4。位元數越低、吞吐越高、記憶體佔用越小。Blackwell 的 Transformer Engine 導入硬體 FP4，是其推論速度暴增的主因。Sparsity（2:1 稀疏）可再讓標稱算力翻倍。

Process Node / 製程

代工製程（TSMC）決定電晶體密度與能效。H100 為 4N、Blackwell 為 4NP、Rubin 進入 N3（3nm）、Feynman 預計 A16（1.6nm）。製程每進一步，同功耗下算力提升。

▸CPU：從 x86 主機到 ARM 協同處理器

AI 機架內的 CPU 角色已從「主機」轉為「為 GPU 餵資料的協同處理器」。NVIDIA 自研 ARM CPU 並以 NVLink-C2C 與 GPU 做記憶體一致連接，取代傳統的 PCIe 主機介面：

  • Grace（Blackwell 世代）72 顆 ARM Neoverse V2 核心，TSMC N4，搭載 480 GB LPDDR5X；以 NVLink-C2C 900 GB/s 與 Blackwell GPU 共享統一記憶體位址空間。
  • Vera（Rubin 世代）88 顆自研 ARM「Olympus」核心 / 176 執行緒，227B 電晶體，Arm v9.2；NVLink-C2C 升級至 1.8 TB/s，可掛載至 1.5 TB LPDDR5X。取代 Grace。
  • x86（Intel Xeon / AMD EPYC）仍是 HGX 機板與多數企業伺服器的主機 CPU，透過 PCIe 連接 GPU。

▸Superchip：CPU + GPU 封裝在一起

把 CPU 與 GPU 用 C2C 綁成單一模組，即為 Superchip。這是 NVL72 等機架的基本積木：

  • GB200 Superchip = 2× Blackwell GPU + 1× Grace CPU，C2C 900 GB/s 連接；整顆模組 TDP 約 2,700 W（含 2 GPU + CPU）。
  • GB300 Superchip = Blackwell Ultra 版本，記憶體與網路（ConnectX-8，800 Gb/s）加倍。
  • Vera Rubin 配對取代 Grace Blackwell；另有 Rubin CPX——專為長上下文 prefill 設計、改用 GDDR7（128 GB）而非 HBM 的特化推論加速器。

02 記憶體層 · HBM

把 DRAM 疊起來，貼到 GPU 旁邊

大模型的真正硬牆往往不是算力，而是「能不能把模型權重與 KV-cache 裝進記憶體、並餵得夠快」。HBM 就是為此而生。

▸原理：把 DRAM 疊起來、貼到 GPU 旁邊

傳統 DRAM 透過 PCB 走線連到處理器，頻寬受限。HBM（High Bandwidth Memory）改變了物理佈局：

  • 3D 堆疊——多顆 DRAM 晶粒垂直堆疊（如 12-Hi），以 TSV（矽穿孔）垂直導通，形成一個「stack」。
  • 2.5D 封裝——HBM stack 與 GPU 晶粒並排坐在同一片矽中介層（interposer）上，用極寬的匯流排（每 stack 1024-bit，HBM4 加倍至 2048-bit）連接。
  • CoWoS（Chip-on-Wafer-on-Substrate）——TSMC 的先進封裝技術，把 GPU die + 多顆 HBM 整合在一個封裝內。CoWoS 產能與良率是當前 AI 晶片的主要供應瓶頸之一。

CoWoS 的側視剖面：多顆 DRAM 晶粒垂直 3D 堆疊（虛線＝TSV 矽穿孔），與 GPU 晶粒並排坐在同一片矽中介層上（2.5D），再一起封進基板。HBM4 的關鍵不是拉高 pin 速率，而是把每 stack 的匯流排從 1024-bit 加倍到 2048-bit。

03 Scale-Up 互連 · In-Rack Fabric

讓一櫃 GPU，表現得像一顆超大 GPU

這是 NVIDIA 最深的護城河。Scale-up fabric 讓一櫃 GPU 表現得像「一顆超大 GPU」，靠的是銅線、而非光纖。

▸NVLink：GPU 對 GPU 的高速通道

NVLink 是 GPU 之間的點對點高速連結，頻寬遠勝 PCIe（PCIe Gen5 ×16 僅約 128 GB/s）。每一代翻倍：

  • NVLink 4（Hopper）每 GPU 900 GB/s
  • NVLink 5（Blackwell）每 GPU 1.8 TB/s，約為 PCIe Gen5 的 14×
  • NVLink 6（Rubin）每 GPU 3.6 TB/s
  • NVLink 7（Rubin Ultra）持續倍增，支撐 576-GPU domain

▸NVSwitch：把所有 GPU 連成 all-to-all

單靠點對點 NVLink 無法讓 72 顆 GPU 兩兩全互連。NVSwitch 是專用交換晶片，做出「任一 GPU 可全速存取任一 GPU 記憶體」的拓撲。在 GB200 NVL72 中，9 個 switch tray 內的 NVSwitch 4.0 提供整櫃 130 TB/s 的 all-to-all 頻寬，且任意 GPU 間延遲低於 300 ns，使整櫃可被當成單一加速器來編程。NVSwitch 4.0 單晶片即達 14.4 TB/s 聚合頻寬。

左：純點對點 NVLink，GPU 一多就兩兩接不完、無法全速全互連。右：NVSwitch 作為交換中樞，讓任一 GPU 全速存取任一 GPU 的記憶體（all-to-all）——GB200 NVL72 用 9 個 switch tray 做出整櫃 130 TB/s、延遲 <300 ns，整櫃因此能被當成「一顆 GPU」來編程。

▸NVLink-C2C 與 NVLink Domain

NVLink-C2C 把同樣的技術用在 CPU↔GPU 之間，做出快取一致（cache-coherent）的統一記憶體——GPU 可直接讀 CPU 的 LPDDR5X、CPU 也可讀 GPU 的 HBM，無需顯式複製。NVLink domain（world size）是 scale-up 的核心指標，指「能以 NVLink 全互連的最大 GPU 數」：

▸競爭標準：UALink 與 Infinity Fabric

AMD 與 Intel、Google、Microsoft 推動開放標準 UALink（Ultra Accelerator Link）對抗封閉的 NVLink。AMD 內部以 Infinity Fabric 做 8-GPU 全互連；MI400 的 Helios 機架採「UALink-over-Ethernet」，scale-up domain 同樣達 72 顆、260 TB/s。但截至 2026 年中，UALink 尚無大規模量產部署，互連成熟度仍是 AMD 在 frontier 級訓練上的主要劣勢。

04 Scale-Out 互連 · Cluster Fabric

把上千個機架，連成一座叢集

當模型大到一個 NVLink domain 也裝不下，就得把成百上千個機架連成叢集。這層走光纖、用 RDMA，是 scale-out 的領域。

▸InfiniBand：HPC 與 turnkey 叢集的主力

InfiniBand（IB）是低延遲、原生支援 RDMA（遠端直接記憶體存取，繞過 CPU）的網路，長年主導超算與 NVIDIA DGX SuperPOD 等整機方案。其獨門武器是 SHARP（In-Network Computing）——把 all-reduce 等集合運算直接在交換器內完成，最高可加速集合通訊達 32×。

  • NDR——每埠 400 Gb/s，Quantum-2 交換器（64×400G），PAM4 調變。
  • XDR——每埠 800 Gb/s，Quantum-X800 交換器（144×800G，Quantum-3 ASIC，整合矽光子）。

▸Ethernet 的反攻：Spectrum-X 與 Ultra Ethernet

超大規模業者偏好以 Ethernet 統一網路。為此 NVIDIA 推出 Spectrum-X——針對 AI 流量優化、跑 RoCE（RDMA over Converged Ethernet）的 Ethernet 平台；Rubin 世代的 Spectrum-6 更採共封裝光學（CPO）。業界則以 Ultra Ethernet（UEC）聯盟標準，把 Ethernet 改造成可與 IB 競爭的 AI fabric。

▸端點：NIC / SuperNIC 與 DPU

  • ConnectX SuperNIC——伺服器端的網卡。ConnectX-7（400G）→ ConnectX-8（800G）→ ConnectX-9（1.6 Tb/s，Rubin 世代）。AMD 對應為 Pensando「Vulcano」/「Pollara」。
  • BlueField DPU——資料處理單元，把網路、儲存、安全、虛擬化的工作從 CPU 卸載。BlueField-4 內含 64 核 Grace CPU + ConnectX-9。

05 系統與機架 · Rack-Scale

把整套硬體，縫合成一台加速器

把晶片、記憶體與兩張網組裝成可運轉的系統，有三種封裝層級——機板、機架、與整套參考設計。

▸三種系統封裝：DGX / HGX / MGX

DGX

NVIDIA 自家的整機方案（turnkey），軟硬體整合、開箱即用，附 NVIDIA 支援。例：DGX B200、DGX SuperPOD。

HGX

8-GPU 基板（baseboard），賣給 OEM（Dell、Supermicro、HPE 等）自行組裝成伺服器。是「一台 8 卡 GPU 伺服器」的標準積木。

MGX

模組化機架參考架構，GB200 NVL72 與 VR200 即衍生自 MGX。NVIDIA 已將其貢獻給 OCP（開放運算計畫），讓生態夥伴共用設計、縮短部署時間達 50%。

▸解剖 GB200 NVL72：一個機架的內臟

NVL72 是當前 AI 機架的範本。它不是「72 台伺服器」，而是一台被液冷與銅纜縫合成的單一加速器：

  • 18× compute tray（1U）——每 tray 含 2 顆 Grace + 4 顆 Blackwell（即 2 個 GB200 Superchip）。
  • 9× NVLink switch tray——內含 NVSwitch 4.0 晶片，構成整櫃 all-to-all fabric。
  • NVLink 銅纜背板——4 個 cartridge、超過 5,000 條同軸銅纜（含 144 條 NVLink 銅纜）。整櫃 scale-up 全靠這片銅背板，是 NVL72 的標誌。
  • 電力子系統——8 個 power shelf（每個 6× 5.5 kW PSU、N+N 冗餘），把 AC 轉成 ~50 V DC，經 busbar（匯流排）分配給各 tray。
  • 液冷歧管（manifold）——把冷卻液送進每個 cold plate，整櫃 72 條冷卻管。

06 網路拓撲 · Topology

怎麼連，和用什麼連一樣重要

把幾千個機架連成叢集時，「怎麼連」和「用什麼連」一樣重要。AI 流量的特性，決定了拓撲不能照搬傳統企業網路。

▸三張獨立的網

一座 AI 叢集通常同時跑三張彼此獨立的 fabric，互不干擾：

  • Compute fabric（運算網）——GPU 間的梯度同步主幹，最高頻寬、最在意延遲，採 rail-optimized fat-tree（IB 或 Spectrum-X）。
  • Storage fabric（儲存網）——獨立於運算網以免互搶頻寬，走 IB 或 RoCE，每節點至少 40 GB/s。
  • Management fabric（管理網）——Ethernet，負責開機佈建、監控、對外存取。

▸三種拓撲

Fat-Tree（胖樹）

NVIDIA 預設拓撲，提供 full bisection bandwidth（任意兩半之間頻寬等於單側總頻寬）。三層架構搭 64 埠交換器可擴展到 32,768 個端點。代價是交換器數量多、成本高。

Rail-Optimized

超大規模 AI 叢集的參考設計。把每顆 GPU 的網卡（一條「rail」）接到不同的 leaf 交換器，讓跨節點通訊只需一跳、並與機內 NVLink 路徑對齊，大幅降延遲。

Dragonfly+

以更少的 hop 數與交換器成本，支撐超過百萬節點的規模，但需要 adaptive routing。適合超大叢集的成本優化。

左 Fat-Tree：脊（spine）與葉（leaf）全連接，提供 full bisection bandwidth——任意兩半之間都能跑滿，代價是交換器多、貴。右 Rail-Optimized：把每顆 GPU 的網卡（一條 rail）接到不同 leaf，跨節點走同一條 rail 只需一跳、並與機內 NVLink 路徑對齊，大幅降延遲——這是超大規模 AI 叢集的參考設計。

SuperPOD 是 NVIDIA 的叢集參考架構（如以 Quantum-X800 連接的 B300 SuperPOD），把上述拓撲、佈線規範、儲存與管理網打包成可複製的藍圖。網路設定錯誤的代價極高——Meta 的研究指出，網路相關問題造成其 GPU 任務失敗的 10.7%。

07 儲存層 · Storage

GPU 再快，餵不飽也是空轉

GPU 再快，餵不飽也是空轉。儲存層的任務是用對的成本，在對的速度把訓練資料與 checkpoint 送到 GPU 面前。

▸四階儲存階層

  • HBM（GPU 內）——最快、最貴、最小。模型權重與 activation 的工作集。
  • 本地 NVMe SSD（節點內）——暫存、資料集快取、spill。
  • 高效平行檔案系統（共享）——餵資料與寫 checkpoint 的主力，需數十～數百 GB/s 聚合吞吐。
  • 物件 / 容量儲存與歸檔——原始資料湖、冷資料，以容量與單位成本為優先。

▸平行檔案系統與 GPUDirect

共享層常見方案：Lustre、IBM Storage Scale（GPFS）、WEKA、VAST Data、DAOS。它們把資料條帶化（striping）到多個儲存節點以堆疊頻寬。關鍵加速技術是 GPUDirect Storage——讓資料從 NVMe / 網路直接 DMA 進 GPU 記憶體，繞過 CPU 與系統記憶體的中轉，消除瓶頸。儲存流量同樣走 IB 或 RoCE，並刻意與 compute fabric 分離，避免在 checkpoint 時與梯度同步互搶頻寬。

08 電力架構 · Power

供電，從基礎建設變成產品本身

當一個機架的功耗從 40 kW 衝到 600 kW，供電就從「基礎建設」變成了「產品本身」。

▸機架功率密度的爆炸

五年內，部署單位從「機房供得起的一個機架」變成「吃掉一座小工廠用電的一個機架」：HPC/Hopper ~40 kW → GB200 ~120 kW → VR200 ~190–230 kW → Kyber ~600 kW，並朝 1 MW 前進。

▸供電鏈：從電網到晶片

傳統 AC 供電鏈為：

變電站 → 變壓器 → 開關設備 → UPS（電池）→ PDU → busway → 機架 power shelf → PSU（AC→DC）→ busbar → tray

每一級都有轉換損耗與冗餘需求（N+1 / N+N）。

▸800 VDC：為了 600 kW 機架的必要轉向

在 Kyber 這種 600 kW 級密度下，傳統 415/480 VAC 的銅排會粗到不可行。NVIDIA 於 2025 年提出 800 VDC 設施對機架的直流供電架構：

  • 同樣截面的銅可多傳輸 150%+ 的電力，每機架省下約 200 kg 銅排。
  • 把整流（AC→DC）集中到 sidecar 電力機架，緊鄰 compute 機架擺放，取代傳統 UPS / PDU / busway 鏈。
  • per-rack 饋線電流從 GB300 的 60 A 跳到 VR200 的 100 A。

給容量規劃的提醒：以一座 10 MW 的 AIDC 為例，若採 GB200（~120 kW/櫃）約可佈 ~80 櫃；若升級到 VR200（~200 kW/櫃）則同樣 10 MW 只能擺 ~50 櫃。電力預算（而非機房面積）才是決定 GPU 數量的真正上限——這也是計費模型裡「每 GPU-hour 攤提」的物理根源。

電力預算，而非機房面積，才是決定 GPU 數量的真正上限。

09 散熱架構 · Cooling

Blackwell 起，液冷不再是選項

Blackwell 起，液冷不再是選項，而是物理上的強制要求。這一層往往是 AIDC 設計裡最被低估、卻最先觸頂的限制。

▸為什麼氣冷已經出局

一個標準氣冷機架的上限約 8–25 kW；而 GB200 NVL72 單櫃就 120 kW+，是氣冷天花板的 5–15 倍。更根本的是熱通量（heat flux）：Blackwell GPU 晶粒達 500–600 W/cm²（接近火箭引擎燃燒室等級），遠超氣冷散熱片所能處理的 5–15 W/cm²。NVIDIA 已正式確認 GB200 NVL72 強制液冷。

▸液冷的幾種形態

Direct-to-Chip (DLC)

當前主流。cold plate（冷板）直接貼在晶片上，內走微流道冷卻液（每模組 ~2–3 L/min、入水溫 30–45°C）。Blackwell / Rubin 機架的標配。

Rear-Door HX (RDHx)

後門式熱交換器，裝在機架背門以水冷散去排出的熱風。適合中等密度或既有機房改造。

Immersion（浸沒）

把整台伺服器泡進絕緣冷卻液，分單相與兩相。散熱效率最高，但對機房設計與維運改動最大。

▸CDU、暖水迴路與 PUE

  • CDU（Coolant Distribution Unit）——把設施側水迴路與機架側技術迴路隔開，是液冷部署的容量設計核心，再經 manifold 把冷卻液分配到各 cold plate。
  • 暖水冷卻（warm-water）——用較高溫的進水即可帶走熱，讓設施能免冰機（free cooling）運轉，大幅降低能耗。
  • PUE（Power Usage Effectiveness）——設施總用電 ÷ IT 用電，越接近 1.0 越好。液冷 + 暖水可把 PUE 壓得比氣冷低很多。VR200 的 compute/switch tray 已完全無風扇，冷卻液流量約為 GB300 的 2×、機架風流需求則降約 80%。
10 成本與 TCO · Economics

最終一切，折算成每 GPU-hour

所有上述硬體，最終會被折算成一個數字：每 GPU-hour 的成本。理解 TCO 的組成，是設計任何 GPU 計費模型的前提。

▸TCO 的兩個構面

CapEx · 資本支出

GPU 本身（單顆 Blackwell ~$30–50k）、一櫃 NVL72 ~$3–4M、網路設備、以及設施興建（百櫃級部署的機房建置可達 $100M）。HBM 與 CoWoS 先進封裝是晶片成本的大頭。

OpEx · 營運支出

電力（最大宗）、冷卻、維運與人力。以單套系統粗估，年營運成本約 $1M：電力 ~$400k、冷卻 ~$100k、設施/維護 ~$500k。

▸從 TCO 推導每 GPU-hour 成本

單位成本（unit cost）的骨架可寫成：

每 GPU-hour 成本 ≈ (攤提 CapEx + 期間 OpEx) ÷ (GPU 數 × 時數 × 利用率)
  • 利用率（utilization）是槓桿——同樣的 CapEx，80% 利用率 vs 40% 利用率，攤到每 GPU-hour 的成本差近一倍。提升利用率（排程、quota、超賣策略）往往比壓低硬體採購價更有效。
  • 電力是 OpEx 主軸——機架功率密度（§08）直接決定電費攤提；這也是為何 cost-per-GPU-hour 與 PUE、kW/櫃 緊密相關。
  • unit cost ≠ unit price——成本是上述攤提；定價則疊加毛利、風險溢價與市場供需。兩者應在計費模型中明確分離。
